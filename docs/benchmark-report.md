# Gwanbase 벤치마크 보고서

**측정일**: 2026-05-05  
**환경**: macOS (Darwin 25.3.0), JVM 17, Gradle JMH Plugin 0.7.2  
**설정**: Warmup 2회, Measurement 5회, Fork 1  
**버퍼 풀 크기**: 1024 프레임 (4MB)

---

## 1. B+Tree (KVStore) 벤치마크

디스크 기반 B+Tree의 단일 스레드 throughput을 측정한다.
KVStore는 SQL 파싱/WAL을 거치지 않고 스토리지 레이어에 직접 접근한다.

### 결과

| 벤치마크 | 사전 로드 | Throughput (ops/s) | 오차 (95% CI) |
|---------|----------|-------------------|--------------|
| insert | 100건 | 76,404 | +/- 21,027 |
| insert | 1,000건 | 73,053 | +/- 21,211 |
| insert | 10,000건 | 71,960 | +/- 23,561 |
| pointLookup (hit) | 100건 | 5,321,753 | +/- 151,031 |
| pointLookup (hit) | 1,000건 | 3,339,417 | +/- 108,752 |
| pointLookup (hit) | 10,000건 | 1,509,505 | +/- 21,232 |
| pointLookup (miss) | 100건 | 7,353,118 | +/- 70,817 |
| pointLookup (miss) | 1,000건 | 3,825,595 | +/- 121,747 |
| pointLookup (miss) | 10,000건 | 1,516,042 | +/- 68,943 |

### 분석

- **삽입 throughput은 데이터 크기에 거의 무관**하다 (~71K-76K ops/s). B+Tree의
  삽입 비용이 O(log N)이므로 100→10,000건에서도 약 6%만 감소한다.
- **검색은 데이터 크기에 로그 비례로 느려진다**. 100건 (트리 높이 2) 대비
  10,000건 (트리 높이 3-4)에서 약 72% 감소.
- **miss가 hit보다 빠르다** (100건 기준 7.35M vs 5.32M). miss는 리프 도달 후
  바로 null 반환하지만, hit는 키 비교 + 값 복사가 추가된다.
- 버퍼 풀 1024 프레임이면 10,000건의 모든 페이지가 메모리에 상주하므로,
  측정값은 순수 자료구조 + BufferPoolManager 오버헤드를 반영한다.

## 2. SQL End-to-End 벤치마크

SQL 문 실행의 전체 경로를 측정한다:
`파싱 → 바인딩 → 옵티마이저 → 연산자 실행 → WAL 로깅 → fsync`

### 결과

| 벤치마크 | 사전 로드 | Throughput (ops/s) | 오차 (95% CI) |
|---------|----------|-------------------|--------------|
| INSERT | 100건 | 249 | +/- 11 |
| INSERT | 1,000건 | 239 | +/- 14 |
| SELECT (점 검색) | 100건 | 246 | +/- 13 |
| SELECT (점 검색) | 1,000건 | 235 | +/- 30 |
| SELECT (필터 스캔) | 100건 | 248 | +/- 10 |
| SELECT (필터 스캔) | 1,000건 | 233 | +/- 38 |
| UPDATE (단건) | 100건 | 248 | +/- 8 |
| UPDATE (단건) | 1,000건 | 235 | +/- 32 |

### 분석

- **모든 SQL 연산이 ~240 ops/s로 균일하다.** INSERT, SELECT, UPDATE 간
  throughput 차이가 거의 없다.
- 이는 **WAL flush (fsync)가 지배적 병목**임을 강하게 시사한다:
  - 각 SQL은 auto-commit 트랜잭션으로 실행되며, 커밋 시 WAL을 디스크에 flush한다.
  - fsync 한 번에 ~4ms가 소요된다고 가정하면 1/0.004 = 250 ops/s와 일치한다.
  - 1,000건 full scan과 점 검색의 차이가 미미한 것이 증거다.
- **데이터 크기 효과**: 100건 → 1,000건에서 약 5% 감소. 이는 연산 자체의
  비용 증가가 아니라 WAL 레코드 크기 증가에 의한 미미한 차이다.

## 3. 병목 분석 요약

```
┌─────────────────────────────────────────────────────────────┐
│  SQL End-to-End (~240 ops/s)                                │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌────────────┐  │
│  │ Parsing  │→ │Optimizer │→ │Execution │→ │ WAL flush  │  │
│  │  ~1 us   │  │  ~1 us   │  │ ~10 us   │  │  ~4000 us  │  │ ← 병목
│  └──────────┘  └──────────┘  └──────────┘  └────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

WAL flush가 전체 레이턴시의 ~99%를 차지한다.

## 4. 성능 개선 로드맵

| 우선순위 | 최적화 | 예상 효과 | 복잡도 |
|---------|--------|----------|--------|
| 1 | **Group Commit** — 여러 트랜잭션의 WAL을 모아서 flush | 10-50x throughput 향상 | 중간 |
| 2 | **읽기 전용 트랜잭션 최적화** — SELECT에서 WAL flush 생략 | SELECT 2x 이상 향상 | 낮음 |
| 3 | **인덱스 활용** — 점 검색에 B+Tree 인덱스 사용 | 대량 데이터 SELECT 향상 | 이미 구현 (ANALYZE 필요) |
| 4 | **버퍼 풀 프리페치** — 순차 스캔 시 미리 읽기 | full scan 향상 | 중간 |
| 5 | **WAL 압축** — full-page image 대신 delta 로깅 | WAL 크기 감소, flush 속도 향상 | 높음 |

## 6. 비교 참고

| 시스템 | 단건 INSERT (ops/s) | 비고 |
|--------|-------------------|------|
| Gwanbase | ~240 | 단일 스레드, 매 커밋 fsync |
| SQLite (WAL mode) | ~1,000-5,000 | Group commit, 경량 WAL |
| PostgreSQL | ~10,000-50,000 | Group commit, 비동기 커밋 옵션 |

Gwanbase의 throughput은 "매 트랜잭션마다 fsync" 정책에서 이론적 상한에 근접한다.
Group commit 도입 시 실질적인 성능 비교가 가능해진다.
